## Custom Project

### Dataset

The producer uses a CSV file named:

- data/sales.csv

This dataset contains simulated e-commerce sales transactions. Each record represents a single order event.

Each row includes fields such as:

- order_id
- customer_id
- region_id
- product_id
- unit_price
- quantity
- currency_code
- device_type
- payment_method
- referral_source
- datetime
- customer_note
- kafka_key
- kafka_partition
- kafka_offset

### Kafka Messages

The producer sends each row from the CSV as a JSON-like Kafka message.

- Kafka topic used: streaming-02-kafka-case
- Message key: customer_id
- Message content: full sales record (row-based dictionary)

The key modification made to the producer was changing the partitioning strategy so messages are keyed by customer instead of region, improving grouping by customer behavior in Kafka partitions.

### Consumer Processing

The consumer reads messages from the same Kafka topic and processes them sequentially.

- It consumes all available messages (3 messages in this run)
- It logs each raw message to the terminal
- It writes processed records to a CSV file: data/output/consumed_sales_modified.csv

The consumer was extended with a transformation step that:

- Computes total_price = unit_price × quantity
- Normalizes region_id into lowercase region_clean
- Adds a lineage field processed_by = consumer_foster_pipeline

No filtering is applied; all messages are processed and stored.

### Experiments

Two main modifications were made:

*Phase 4 (Producer modification):*

- Changed Kafka message keying from region_id to customer_id
- This changes how messages are distributed across Kafka partitions

*Phase 5 (Consumer enhancement):*

- Added a transformation function (process_message)
- Introduced derived fields (total_price, region_clean)
- Added data lineage tracking (processed_by)

### Results

When running the pipeline:

- The producer successfully sent 3 messages to Kafka.
- The consumer consumed all 3 messages from the topic.
- Each message was processed and enriched before being written to CSV.
- A new CSV file was generated containing both original and derived fields.

The final output file: data/output/consumed_sales_modified.csv

### Interpretation

This Kafka streaming workflow demonstrates how raw event data can be transformed into analytics-ready data in real time.

Key changes from the original example include:

- Moving from raw message consumption to enriched data processing
- Adding computed fields for business analysis
- Changing message partitioning strategy based on customer instead of region

From a business perspective, this pipeline could be used to:

- Track customer spending behavior (total_price)
- Analyze regional performance trends (region_clean)
- Trace data lineage for debugging and auditing (processed_by)

Overall, the system shows how streaming data can be converted into structured datasets suitable for downstream business intelligence, reporting, or machine learning workflows.
